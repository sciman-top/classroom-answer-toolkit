# 规则快照与资产协议设计

## 1. 文档定位

本文档是 [工程终态与平台化路线图.md](D:/CODE/初中物理试卷参考答案生成与渲染/工程终态与平台化路线图.md) 的实现级配套文档，聚焦：

- 规则资产如何组织
- 规则如何版本化
- 规则如何编译为运行时快照
- 工具如何统一消费快照
- 规则变化如何做到灵活、可复现、可回滚

本文档不重复讲产品终态，只定义规则与资产协议。

## 2. 设计目标

### 2.1 目标

规则系统必须同时满足：

- 灵活变化
- 不被固定字段写死
- 规则来源可追踪
- 同一结果可回放
- 工具层统一口径
- 学科可扩展

### 2.2 反目标

不采用以下做法：

- 工具直接读取大规格 md 作为运行时输入
- 每个工具各自维护一套规则解释逻辑
- 遇到特殊需求直接改主规则文件
- 新规则只写文档、不进结构化资产

## 3. 资产分层

### 3.1 平台核心资产

建议路径：

- `prompts/platform-core/manifest.json`
- `prompts/platform-core/rules/*.json`
- `prompts/platform-core/profiles/*.json`
- `prompts/platform-core/migrations/*.json`

用途：

- 承载跨学科稳定不变量
- 不承载学科特有口径

平台核心规则示例：

- 交付物只保留 `.md` 和 `.pdf`
- 数学公式必须真实渲染
- 视觉审阅是发布门禁
- profile 继承机制

### 3.2 学科包资产

建议路径：

- `prompts/physics-answer/manifest.json`
- `prompts/physics-answer/rules/*.json`
- `prompts/physics-answer/profiles/*.json`
- `prompts/physics-answer/migrations/*.json`
- `prompts/physics-answer/spec.md`
- `prompts/physics-answer/checklists/acceptance.md`

用途：

- 承载学科专有口径
- 承载题型规则
- 承载课堂展示要求
- 承载学科 profile override

### 3.3 共享协议资产

建议路径：

- `prompts/shared/schemas/manifest.schema.json`
- `prompts/shared/schemas/rule-pack.schema.json`
- `prompts/shared/schemas/rule.schema.json`
- `prompts/shared/schemas/profile.schema.json`
- `prompts/shared/schemas/override.schema.json`
- `prompts/shared/schemas/snapshot.schema.json`

用途：

- 对结构做约束
- 不把字段封死
- 保证未来新增字段可兼容

### 3.4 评测资产

建议路径：

- `eval/<subject-pack>/dataset.json`
- `eval/<subject-pack>/cases/*`
- `eval/<subject-pack>/baselines/visual/*`
- `eval/<subject-pack>/results/*`

用途：

- 保障规则变化可回归
- 保障渲染行为可回归
- 保障视觉门禁可回归

## 4. 核心对象模型

### 4.1 SubjectPackManifest

用途：

- 描述学科包身份、版本、依赖、入口和评测绑定

建议字段：

- `schemaVersion`
- `assetId`
- `displayName`
- `version`
- `status`
- `locale`
- `domain`
- `kind`
- `inherits`
- `sourceOfTruth`
- `evaluation`
- `tooling`

说明：

- `kind` 用于标记 `platform-core`、`subject-pack`、`profile-pack`
- `inherits` 明确上游依赖，不允许隐式继承

### 4.2 RulePack

用途：

- 承载一组规则，而不是把所有规则塞进一个大配置文件

建议字段：

- `schemaVersion`
- `packId`
- `packVersion`
- `kind`
- `rules`
- `metadata`

### 4.3 Rule

Rule 采用“固定骨架 + 扩展载荷”的形式。

固定骨架建议：

- `id`
- `version`
- `enabled`
- `severity`
- `priority`
- `engine`
- `appliesTo`
- `params`
- `source`
- `notes`
- `extensions`

含义：

- `id`：规则唯一标识
- `version`：单条规则版本
- `enabled`：规则开关
- `severity`：`hard` / `soft` / `hint`
- `priority`：冲突解决顺序
- `engine`：指向由哪个子系统消费
- `appliesTo`：作用范围
- `params`：规则参数
- `source`：来源追踪
- `extensions`：保留未来字段扩展

### 4.4 RenderProfile

Profile 只管展示与排版，不承载学科推理。

建议字段：

- `name`
- `inherits`
- `layout`
- `graphics`
- `review`

`layout` 建议包含：

- `plainTextCjkTarget`
- `plainTextCjkMax`
- `bodyFontSizePt`
- `lineHeight`
- `pageMargins`
- `answersPerLine`

`graphics` 建议包含：

- `maxFigureWidthMm`
- `minReadableFigureHeightMm`
- `captionFontSizePt`
- `multiFigurePacking`

### 4.5 Override

Override 只用于显式特殊要求。

建议字段：

- `overrideId`
- `baseSnapshot`
- `reason`
- `author`
- `changes`

`changes` 的操作类型建议：

- `replace`
- `patch`
- `disable`
- `append`

### 4.6 ResolvedSnapshot

运行时唯一真相。

建议字段：

- `snapshotId`
- `generatedAt`
- `layers`
- `subjectPack`
- `activeProfile`
- `rules`
- `profiles`
- `toolchain`
- `inputRefs`
- `meta`

要求：

- 渲染器、校验器、评测器只读 `ResolvedSnapshot`
- 不允许工具绕过快照再读取散落规则

## 5. 规则分类

### 5.1 稳定不变量

适合进入平台核心层：

- 交付物限制
- 数学渲染必须真实公式
- 视觉审阅门禁
- profile 继承规则
- 快照记录与复现要求

### 5.2 可变规则

适合进入学科包和 profile：

- 行宽
- 题型布局
- 选择题每行数量
- 非选择题简洁度策略
- 单页优先阈值
- 作图题默认排版策略

### 5.3 仅文档化规则

不宜立刻硬结构化：

- 尚未稳定的教研表述
- 模糊经验性建议
- 需要大量人工判断的审题提醒

处理方式：

- 保留在 `spec.md`
- 在 `Rule.source` 中引用来源
- 必要时以 `notes` 或 `hint` 形式挂入规则系统

## 6. 规则优先级与冲突解决

### 6.1 合并顺序

固定顺序：

1. `platform-core`
2. `subject-pack`
3. `profile-pack`
4. `task-override`
5. `hotfix-override`

### 6.2 冲突解决策略

规则冲突时：

1. 先按层级覆盖
2. 同层内按 `priority` 覆盖
3. 同 `id` 同 `priority` 视为资产错误，编译失败

### 6.3 强度策略

- `hard`：违背则失败
- `soft`：生成告警
- `hint`：只做提示

用途：

- 避免把所有要求都做成“非黑即白”
- 保留演进过程中的灰度空间

## 7. 版本策略

### 7.1 Pack 版本

采用：

- `major`
- `minor`
- `patch`

规则：

- `major`：破坏性协议变更
- `minor`：新增规则、字段、profile、eval
- `patch`：阈值与默认值微调

### 7.2 单条规则版本

单条规则也允许带 `version`：

- 用于审计某条规则是否发生语义变化
- 用于将“规则变更”与“pack 整体变更”区分开

### 7.3 Migration

每次 `minor/major` 升级建议配 migration 说明：

- `from`
- `to`
- `changes`
- `compat`
- `actions`

## 8. Snapshot 编译流程

### 8.1 输入

输入固定为：

- 平台核心 manifest 与规则包
- 学科包 manifest 与规则包
- 生效 profile
- 显式 override
- 工具版本信息

### 8.2 流程

1. 校验所有资产 schema
2. 加载继承链
3. 读取平台规则
4. 读取学科规则
5. 读取 profile
6. 读取 override
7. 合并规则
8. 解析冲突
9. 产出最终 profile 参数
10. 注入工具链版本
11. 生成 `snapshotId`
12. 写出 `ResolvedSnapshot`

### 8.3 输出

建议写出到：

- `.snapshot-cache/<snapshotId>.json`

同时在交付清单中记录：

- `snapshotId`
- `snapshotPath`

## 9. Validator 接入设计

### 9.1 现状问题

当前 validator 仍是“直接读 profile / config”。

风险：

- 渲染器与校验器可能读到不同口径
- 规则变更无法精确追踪到某次交付

### 9.2 终态要求

validator 必须只读 `ResolvedSnapshot`。

validator 应支持：

- 资产前置校验
- Markdown 内容校验
- 版式规则校验
- 课堂展示规则校验
- 图文布局规则校验

### 9.3 拆分方案

建议拆成两类：

- `validate-assets`
- `validate-answer`

`validate-assets`：

- 校验 manifest
- 校验 schema
- 校验规则冲突

`validate-answer`：

- 校验 Markdown 与快照是否一致
- 校验题号与首小问同行
- 校验选择题布局
- 校验长行与非选择题简洁度
- 校验公式渲染前置条件

## 10. 渲染器与交付器接入设计

### 10.1 RenderProvider

渲染器只读：

- `snapshot.rules`
- `snapshot.profiles.resolved`

不再直接读取：

- 散落的 root 规格文档
- 工具本地私有常量

### 10.2 Deliver 流程

交付流程固定为：

1. `validate-assets`
2. `compile-snapshot`
3. `validate-answer`
4. `render`
5. `review`
6. `cleanup`
7. `write-delivery-manifest`

## 11. 评测接入设计

### 11.1 EvalSuiteManifest

评测集应绑定：

- 学科包版本
- 规则快照版本
- baseline 版本

### 11.2 最低覆盖

必须覆盖：

- smoke
- 选择题紧凑布局
- 题号与首小问同行
- 长行控制
- 非选择题去讲题稿化
- 公式严格渲染
- 视觉首屏回归

### 11.3 规则变化后的动作

每次规则变化都要同步：

1. 更新结构化资产
2. 更新对应 eval case
3. 更新必要 visual baseline
4. 记录 migration

## 12. 仓库落地方案

### 12.1 第一批落地

- 将 `v8.8` 同步到 `manifest/config/spec/checklist`
- 新增 `prompts/shared/schemas/`
- 新增 `tools/rule-compiler/`
- 将平级 `formatRules` 逐步迁成 `rules[]`

### 12.2 第二批落地

- validator 改为读取 snapshot
- render / deliver 改为读取 snapshot
- eval 与 snapshot 绑定

### 12.3 第三批落地

- 平台核心资产拆出 `platform-core`
- 物理资产升级为正式 `subject-pack`
- 为第二学科预留空学科包模板

## 13. 风险与回滚

风险：

- 字段过早写死
- schema 太严格导致变更成本过高
- 快照逻辑过重拖慢开发
- 规则过多变成维护负担

控制方式：

- 大部分 schema 允许 `additionalProperties`
- 规则骨架固定，业务参数放进 `params`
- 先只硬化高价值规则
- 逐步迁移，不一次性全量重构

回滚：

- 快照回滚
- Pack 版本回滚
- 工具版本回滚

## 14. 结论

规则快照体系的核心不是“把规则写死”，而是：

- 用固定协议承载变化
- 用快照固化某一时刻的真实规则
- 用评测和视觉基线保证变化不失控

这套设计是后续桌面壳、跨学科扩展、作图题答案图系统能否稳定落地的前置基础。
