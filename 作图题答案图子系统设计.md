# 作图题答案图子系统设计

## 1. 文档定位

本文档是 [工程终态与平台化路线图.md](D:/CODE/初中物理试卷参考答案生成与渲染/工程终态与平台化路线图.md) 的实现级配套文档，专门回答：

- 能否在原题图基础上生成答案图
- 如何把答案图插入答案 PDF 中的合理位置
- 如何兼顾课堂可视度、空间利用率、人工复核与结果可复现

本文档结论：

- 能做
- 但不能走“通用 AI 直接画图”的路线
- 必须做成“结构化叠加 + 可复核 + 可排版”的独立子系统

## 2. 为什么这是最大难题之一

作图题的难点不在“会不会画红线”，而在于以下五类问题同时存在：

1. 语义理解
2. 图像定位
3. 叠加渲染
4. 版式安排
5. 人工兜底

### 2.1 语义理解难

同样叫“作图题”，实际可能是：

- 杠杆力臂题
- 光路题
- 受力分析题
- 电路图连接题
- 电路实物图连线题
- 数学辅助线题
- 几何证明辅助图题

这些题不只是“画一条线”，而是：

- 画在哪里
- 线是什么性质
- 线是否需要延长
- 是否需要垂线符号
- 是否需要箭头和方向
- 是否要文字标签

### 2.2 定位难

系统必须知道：

- 原图边界
- 图中关键点
- 组件位置
- 图中已有线段和已有标注
- 哪些区域不能压住

### 2.3 叠加难

叠加层必须满足：

- 看起来像答案图
- 不破坏原图阅读
- 不挡住关键信息
- 投屏清晰
- 打印仍清晰

### 2.4 排版难

答案图不是单独附件，而是答案块的一部分。

必须解决：

- 图文同页
- 多图拼版
- 空白利用
- 分页控制
- 图注与题号关系

### 2.5 兜底难

完全自动化在高风险作图题上不现实。

因此系统必须内建：

- 人工复核
- 微调
- 局部重画
- 降级交付策略

## 3. 子系统目标

### 3.1 必达目标

系统应支持在原题图基础上生成答案图，并：

- 与答案文字一致
- 在 PDF 中合理排版
- 适合课堂投屏
- 支持人工复核
- 支持结果回放

### 3.2 首批支持场景

建议按难度分三类推进。

第一类，优先落地：

- 杠杆力臂
- 光路
- 受力分析
- 电路连接
- 电路实物图连线

第二类，中期推进：

- 数学几何辅助线
- 凸透镜成像作图
- 磁感线方向补画

第三类，后期推进：

- 高复杂实验装置图
- 多层重叠几何作图
- 大段批改式图面推理

## 4. 总体架构

### 4.1 五段式链路

作图题答案图子系统建议采用五段式：

1. Source Extraction
2. Graphic Understanding
3. Answer Graphic Planning
4. Overlay Rendering
5. Placement & Review

### 4.2 Source Extraction

职责：

- 从原卷 PDF 或图片中裁出题图区域
- 保留原图分辨率和坐标
- 输出统一题图对象

输入：

- 原卷 PDF
- review 页图
- 页内题号定位信息

输出：

- `ProblemFigureAsset`

### 4.3 Graphic Understanding

职责：

- 识别关键点、线段、元件、节点、箭头、标签
- 识别图中已有结构
- 生成可供后续规划的图元基础层

输出：

- `FigureUnderstandingResult`

### 4.4 Answer Graphic Planning

职责：

- 根据题型和答案语义生成结构化作图指令
- 决定画哪些图元、位置、文字、方向

输出：

- `AnswerGraphicSpec`

### 4.5 Overlay Rendering

职责：

- 将结构化图元渲染为叠加层
- 与原图合成
- 生成可用于 HTML/PDF 的图块

输出：

- `AnswerGraphicArtifact`

### 4.6 Placement & Review

职责：

- 把答案图块排进答案正文附近
- 控制尺寸、分页、图注
- 提供人工复核与微调能力

输出：

- `PlacedAnswerGraphic`
- 最终交付中的图文块

## 5. 核心对象模型

### 5.1 ProblemFigureAsset

表示从原卷中抽取出的题图。

建议字段：

- `figureId`
- `sourceDocument`
- `pageNumber`
- `questionRef`
- `imagePath`
- `cropBox`
- `pixelSize`
- `dpi`

### 5.2 FigureUnderstandingResult

表示题图被解析后的可操作结构。

建议字段：

- `anchors`
- `segments`
- `curves`
- `regions`
- `labels`
- `components`
- `confidence`

说明：

- `anchors` 表示关键点
- `components` 对电路、仪器类题图尤为重要

### 5.3 AnswerGraphicSpec

这是子系统最核心的数据模型。

建议字段：

- `graphicId`
- `baseFigureId`
- `intent`
- `primitives`
- `style`
- `caption`
- `placementHints`
- `reviewHints`

### 5.4 Primitive

建议支持的图元类型：

- `line`
- `polyline`
- `arrow`
- `ray`
- `arc`
- `circle`
- `point`
- `right-angle-mark`
- `highlight`
- `label`
- `wire-path`
- `region-mask`

### 5.5 AnswerGraphicArtifact

表示已经渲染好的图块。

建议字段：

- `artifactId`
- `graphicId`
- `baseImagePath`
- `overlaySvgPath`
- `compositedPreviewPath`
- `widthPx`
- `heightPx`
- `boundingBoxes`

### 5.6 PlacedAnswerGraphic

表示进入答案版面的图块。

建议字段：

- `artifactId`
- `questionRef`
- `placementMode`
- `targetBlock`
- `figureWidthMm`
- `captionMode`
- `pageBreakPolicy`

## 6. 推荐实现策略

### 6.1 不直接生成位图答案图

不建议直接输出：

- 一张不可编辑的红笔图
- 一张由通用图像模型生成的图

建议输出：

- 原图
- 结构化叠加层
- 合成预览

原因：

- 可修改
- 可审计
- 可回放
- 可复用到不同 profile

### 6.2 SVG 作为中间表示

建议使用 SVG 作为答案图叠加层的中间表示。

原因：

- 适合线条、箭头、标签
- 可缩放
- 可嵌入 HTML
- 可被 Chromium 高质量打印到 PDF

### 6.3 坐标系统一

所有图元都必须落在统一坐标系上。

推荐坐标策略：

- 以题图裁剪区域左上角为原点
- 使用浮点像素坐标
- 保存原始像素尺寸
- 渲染时再统一做缩放

### 6.4 样式统一

答案图的风格必须统一，不允许不同题像不同软件做出来的。

首版统一样式建议：

- 主标注色：红色
- 辅助色：深蓝或深灰
- 线宽：按 profile 分级
- 标签字体：与正文一致或近似
- 箭头尺寸：按图块尺寸自动缩放

## 7. 自动化策略

### 7.1 以模板驱动为主

首版不追求“任意题图通吃”，而是按题型模板推进。

模板示例：

- `lever-force-arm`
- `optics-ray`
- `free-body-diagram`
- `circuit-schematic-wire`
- `circuit-real-object-wire`

### 7.2 以结构推断为主

推断逻辑建议是：

- 先判题型
- 再选模板
- 再绑定原图 anchors
- 再生成 `AnswerGraphicSpec`

### 7.3 人工兜底为必须

自动结果必须支持以下人工操作：

- 拖动图元
- 删除图元
- 新增图元
- 编辑文字
- 调整颜色和线宽
- 确认图块大小和位置

## 8. 排版设计

### 8.1 图块是答案块的一部分

答案图不应作为附录或页尾附件处理。

建议：

- 就近放在对应题号下
- 与答案文字同容器排版
- 图和关键说明尽量保持在同页

### 8.2 图块尺寸策略

首版建议采用三档：

- `inline-small`
- `inline-medium`
- `full-width`

判定依据：

- 原图复杂度
- 标签数量
- 是否有多条叠加线
- 课堂后排可视度

### 8.3 多图拼版策略

若同一题有多个小图：

- 可横向拼版
- 也可纵向堆叠

但必须保证：

- 图块边界清晰
- 题号对应关系清晰
- 不因拼版导致标签过小

### 8.4 空间利用策略

应充分利用空间，但不允许：

- 图块压得过小
- 图文互相遮挡
- 跨页断裂难读

推荐做法：

- 先按题块估算空间
- 再选择图块模式
- 最后交给排版引擎和 visual review 验收

## 9. 质量门禁

### 9.1 结构正确性

必须检查：

- 图元是否落在正确坐标
- 线条是否连接正确对象
- 垂线是否真的垂直
- 光路方向是否正确
- 电路连线路径是否正确

### 9.2 视觉可读性

必须检查：

- 线宽是否足够
- 标签是否清晰
- 红色叠加是否盖住原图信息
- 投屏时后排是否能看清

### 9.3 版式合理性

必须检查：

- 图块是否就近
- 图文是否同页
- 多图是否拼版合理
- 是否出现断裂分页

### 9.4 交付一致性

必须检查：

- 图中标注与答案文字口径一致
- Markdown 与 PDF 中图块一致
- 图块来源与快照记录一致

## 10. 评测体系

### 10.1 样例库

需要单独建设：

- 杠杆题图样例
- 光路题图样例
- 受力分析样例
- 电路图样例
- 电路实物图样例

### 10.2 评测维度

建议四类评测：

- 图元正确性
- 视觉回归
- 排版回归
- 人工微调可用性

### 10.3 视觉基线

每类题型至少保留：

- 原图基线
- 叠加后基线
- PDF 页面基线

## 11. 技术路线建议

### 11.1 当前阶段

推荐直接基于现有工具链扩展：

- 原卷处理：PDF.js + 当前页图 review 链
- 图像处理：OpenCV
- 叠加层：SVG
- 渲染：HTML + Chromium PDF

### 11.2 桌面复核

WPF 侧首版建议使用：

- Canvas
- Path
- Line
- TextBlock
- Adorner / overlay layer

首版不强求复杂编辑器，只要能：

- 看到图
- 拖动图元
- 编辑标签
- 保存修改

### 11.3 后续增强

后续可考虑：

- SkiaSharp
- 更复杂的图元编辑器
- 半自动 anchor 校正
- 更强的数学几何推断

## 12. 实施计划

### 第一阶段：数据模型与样例

- 定义 `ProblemFigureAsset`
- 定义 `FigureUnderstandingResult`
- 定义 `AnswerGraphicSpec`
- 建立首批固定样例库

### 第二阶段：原型链路

- 实现题图裁剪
- 实现 SVG 叠加层
- 实现合成预览
- 实现 HTML 中嵌图

### 第三阶段：排版与评测

- 实现图块 placement 策略
- 实现 PDF 中就近排版
- 建立视觉基线
- 建立专项评测

### 第四阶段：人工复核

- WPF 中实现最小复核界面
- 支持拖动、删改、增补
- 支持保存为 override 结果

### 第五阶段：跨学科验证

- 在数学几何作图题中试接一类场景
- 验证模型是否足够通用

## 13. 风险与降级

风险：

- 自动定位不准
- 模板覆盖不够
- 数学几何类图复杂度过高
- PDF 中图文排版失衡

降级策略：

- 无法自动作图时，退化为原图 + 文字说明
- 自动结果置信度低时，强制人工复核
- 排版过挤时，切换更宽图块模式

## 14. 结论

“在原题图基础上生成答案图，再合理排进答案 PDF”不是额外锦上添花功能，而是本平台未来最关键的壁垒能力之一。

它必须：

- 独立建模
- 独立评测
- 独立门禁
- 独立进入路线图

只有这样，平台才有机会真正覆盖物理、数学等学科中的高价值作图题场景。
