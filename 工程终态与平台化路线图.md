# 工程终态与平台化路线图

## 1. 文档定位

本文档是本仓当前唯一的工程终态、平台化架构、技术栈、路线图、实施计划、任务清单总纲。

约束：

- 学科业务规范仍以根目录最新版人类规范文件为准。
- 运行时规则、评测、渲染、视觉门禁以结构化资产和编译快照为准。
- 后续新增的架构、路线图、实施类文档，均应引用本文档，不再平行维护第二套总纲。

## 2. 当前现状判断

当前仓库的真实形态，不是桌面产品，而是“提示词/规则资产 + 渲染链 + OCR 工具链”的引擎雏形。

已有基础：

- `prompts/physics-answer/` 已形成资产根。
- `tools/latex-renderer/` 已具备 Markdown + LaTeX -> PDF 的本地链路。
- `tools/ocr/` 已具备本地 OCR 批处理能力。
- `eval/physics-answer/` 已具备 smoke 级回归基础。
- 课堂展示导向、选择题紧凑排布、非选择题去讲题稿化等要求已经进入规范。

当前缺口：

- 人类规范版本与资产层版本仍可能漂移。
- 规则目前仍偏“半结构化”，新增规则容易继续堆进大规格文档。
- 工具层、规则层、评测层尚未完全围绕统一快照运行。
- 还没有正式的桌面产品壳层、安装交付形态和运行时诊断体系。
- 尚未形成跨学科平台 contract。

结论：

- 当前方向正确，但还不是终态。
- 终态应是“Windows 桌面壳 + 本地优先生成引擎 + 学科包体系 + 规则快照体系 + 视觉审阅门禁”。

## 3. 最终工程终态

### 3.1 目标产品形态

最终产品采用“双层终态”：

1. 上层是 Windows 教师桌面应用。
2. 下层是本地优先的生成引擎平台。

上层负责：

- 项目与任务管理
- 原卷导入与页图审阅
- 运行状态、错误、降级提示
- 人工复核与微调
- 交付物浏览与导出

下层负责：

- OCR
- 题号与题型结构识别
- 规则编译
- 答案生成
- 作图题答案图生成
- Markdown 校验
- PDF 渲染
- 视觉审阅
- 交付清理

### 3.2 平台目标

平台终态不是“初中物理专用工具”，而是：

- 平台核心无学科
- 学科能力通过 `subject pack` 接入
- 版式能力通过 `profile` 接入
- 特殊要求通过 `override` 接入
- 运行结果通过 `snapshot` 固化

### 3.3 终态价值

终态要同时满足：

- 课堂可用性优先
- 后排可视度优先
- 公式真实 LaTeX 渲染
- 视觉审阅可验收
- 结果可回放、可审计、可回滚
- 全学段、全学科可扩展

## 4. 架构分层

### 4.1 桌面层

推荐技术栈：

- WPF
- .NET 10
- Generic Host
- CommunityToolkit.Mvvm
- Microsoft.Extensions.DependencyInjection
- Microsoft.Extensions.Options
- Microsoft.Extensions.Logging

桌面层边界：

- 不承接学科规则
- 不承接 OCR 算法
- 不承接渲染核心逻辑
- 只负责 orchestration、review、diagnostics、delivery

### 4.2 平台核心层

平台核心负责定义跨学科 contract：

- `SubjectPackManifest`
- `RulePack`
- `RenderProfile`
- `ResolvedSnapshot`
- `EvalSuiteManifest`
- `DeliveryManifest`
- `OcrProvider`
- `GenerationProvider`
- `RenderProvider`
- `ReviewProvider`
- `AnswerGraphicProvider`

### 4.3 学科包层

每个学科包独立承载：

- 学科规则

- 风险样例
- 评测样例
- 视觉基线
- profile override

当前仓库的 `physics-answer` 应升级为第一个正式 `subject pack`。

### 4.4 工具实现层

推荐保留当前成熟方向：

- 渲染：Markdown-It + KaTeX + Playwright `page.pdf()`

- OCR 辅助链：Tesseract.js
- Windows 安装版可选增强：Windows AI OCR / Windows.Media.Ocr

### 4.5 资产与评测层

资产层：

- 人类规范
- 结构化规则
- profile
- override
- migration

评测层：

- dataset
- fixed cases
- expected outputs
- visual baselines
- local results

## 5. 规则演进架构

### 5.1 设计目标

规则会持续变化，因此不能把 `v8.8` 当固定终态规则。

规则架构目标：

- 可扩展
- 可覆盖
- 可追踪
- 可复现
- 可回滚

### 5.2 四层规则模型

1. 平台基础规则
2. 学科包规则
3. profile 规则
4. 任务 override 规则

合并顺序固定：

1. `platform-core`
2. `subject-pack`
3. `profile-pack`
4. `task-override`
5. `hotfix-override`

### 5.3 规则强度

每条规则必须标记强度：

- `hard`
- `soft`
- `hint`

含义：

- `hard` 不满足即失败
- `soft` 产生告警，但允许继续
- `hint` 只做提示

### 5.4 编译快照

运行时不直接消费大规格文档，只消费 `ResolvedSnapshot`。

快照内容固定包括：

- 规则来源层
- 最终规则列表
- 生效 profile
- 工具链版本
- 评测集版本
- 覆盖层信息
- 快照哈希

### 5.5 复现与回滚

每次交付都记录：

- `snapshotId`
- `subjectPackVersion`
- `profile`
- `overrideId`
- `rendererVersion`
- `ocrProviderVersion`
- `evalDatasetVersion`

回滚策略：

- 切回旧快照
- 切回对应工具版本
- 不手工逆改人类规范正文

## 6. 作图题答案图系统

### 6.1 这是最大的难题之一

是，而且这不是“把图放进 PDF”这么简单。

真正的难点在于：

1. 理解题目到底要求画什么
2. 准确定位原图中的几何位置
3. 在不破坏原图可读性的前提下叠加答案图
4. 让结果既像老师红笔批注，又足够工整、统一、可打印
5. 在 PDF 中合理排版，充分利用空间，但不影响课堂后排可视度

结论：

- 这是平台化过程中最难、最值钱、也最应该单独建子系统的一块。
- 它不能靠通用图像生成胡乱“画图”，必须走“原图 + 结构化叠加 + 人工可复核”的路线。

### 6.2 终态目标

对作图题，系统需要支持：

- 在杠杆图上标出支点、动力臂、阻力臂、垂线、方向
- 在光路图上补作入射光线、折射光线、反射光线、法线
- 在受力图上补作力的方向、作用点和文字标签
- 在电路图或实物图上补连线、开关、电表接法、滑动变阻器接法
- 在数学几何图上补辅助线、角标、垂足、中点、圆弧、阴影、作图痕迹
- 将答案图嵌入答案正文附近，并控制大小、位置、留白和分页

### 6.3 推荐实现路线

作图题答案图不直接输出“AI 生成图片”，而输出“结构化叠加结果”。

推荐链路：

1. 从原卷 PDF 或图片中裁出题图区域
2. 建立题图坐标系
3. 生成结构化作图指令
4. 将作图指令渲染为矢量叠加层
5. 与原题图合成为答案图块
6. 在 Markdown/PDF 中嵌入答案图块
7. 进入人工复核与微调

### 6.4 核心数据模型

新增 `AnswerGraphicSpec`：

- `baseImage`
- `viewport`
- `anchors`
- `primitives`
- `style`
- `caption`
- `placementHints`

其中 `primitives` 支持：

- `line`
- `polyline`
- `arrow`
- `arc`
- `circle`
- `point`
- `label`
- `highlight`
- `brace`
- `region`
- `wirePath`

示例含义：

- 杠杆力臂不是一张新图，而是“基于原图，在某两点之间画垂线和标注”
- 电路实物图连线不是重画整图，而是“在原图上叠加一条红色线段路径”

### 6.5 自动化策略

自动化采用“三段式”：

1. 检测
2. 推断
3. 复核

检测层：

- 识别题图边界
- 识别关键点、元件、节点、文字框
- 识别图中已有线段、圆弧、接线柱、箭头

推断层：

- 根据题型模板和答案语义，推断应该画什么
- 例如杠杆题优先输出“支点到力作用线垂直距离”
- 例如电路实物图优先输出“从电源正极到开关到用电器到负极”的连线路径

复核层：

- 自动结果必须进入人工复核
- 人工可以拖动、删改、增补、微调
- 没有人审的作图结果不作为高置信正式交付

### 6.6 推荐技术路线

推荐技术拆分：

- 原图裁剪与坐标：PDF.js / 当前 review 链路 + OpenCV
- 图元模型：JSON `AnswerGraphicSpec`
- 矢量叠加：SVG 作为中间表示
- 最终输出：HTML 中内联 SVG 或 base image + SVG overlay，再经 Chromium 打印为 PDF
- 桌面复核：WPF Canvas 首版即可，后续如需性能优化再考虑 SkiaSharp

不建议：

- 直接用通用文生图/图生图生成所谓“答案图”
- 直接在位图上不可逆涂抹
- 没有结构化图元、只有一张合成红笔图

### 6.7 排版与空间利用

答案图应被视为“答案块”的一种，而不是附件。

排版策略：

- 默认就近放置在对应题目之后
- 短题尽量图文同页
- 图块优先横向压缩空白，不牺牲后排可视度
- 图块与文字说明共用同一题目容器
- 多个小图可拼版，但必须保证每个图块边界和题号清晰

推荐 profile 参数：

- 图块最大宽度
- 图块最小可视尺寸
- 图文间距
- 图注字号
- 多图拼版阈值
- 单页图块总占比上限

### 6.8 质量门禁

作图题答案图必须新增专项门禁：

- 坐标是否对齐原图
- 红色叠加层是否压住关键原图信息
- 线宽与字体是否适合投屏
- 是否错误连接电路或画错垂线
- 是否与答案文字口径一致
- 是否在 PDF 中分页断裂

### 6.9 结论

作图题答案图系统应被视为平台核心能力，不是附属优化。

它应单独立项，单独评测，单独建立样例库和视觉基线。

## 13. 验收标准

平台终态验收必须同时满足：

- 最新规范与资产层无版本漂移
- 规则变更能生成可追踪快照
- 任一交付结果可按快照复现
- 课堂展示版稳定通过视觉审阅
- 作图题答案图可在原图基础上稳定叠加并排版进 PDF
- 第二学科接入时无需推倒核心 contract

## 14. 风险与回滚

主要风险：

- 规则结构化过早写死
- 学科抽象过度设计
- 作图题自动化准确率不足
- 浏览器和 OCR 运行时环境不稳定
- 视觉门禁覆盖不够

回滚策略：

- 规则问题回滚到旧 snapshot
- 工具问题回滚到旧工具版本
- 作图题子系统不稳定时可降级为“原图 + 文字说明 + 人工补图”
- 新学科接入失败时回到平台 contract 层修正，而不是污染学科包

## 15. 结论

本仓的真正终态不是“更长的提示词”，而是：

- 可演进规则平台
- 可复现生成引擎
- 可审核视觉链路
- 可叠加答案图的题图系统
- 可扩展到全学段、全学科的桌面产品

其中，作图题答案图系统是最大难题之一，也是平台化成败的关键能力之一，应在路线图中单独立项，而不是作为渲染细节附带处理。
