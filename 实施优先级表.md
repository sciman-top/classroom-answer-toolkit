## 2. 实施总原则

先后顺序固定为：

1. 先对齐规范与资产
2. 先建立规则快照
3. 再改校验器和交付器
4. 再扩展评测与视觉基线
5. 最后做作图题子系统和桌面壳

原因：

- 如果规则入口没有统一，后续所有模块都会继续漂移
- 如果 validator 和 deliver 不先统一，作图题和桌面壳都没有稳定依托

## 3. 第一优先级：先建的 3 个目录

### 目录 1

`prompts/platform-core/`

目的：

- 承接跨学科稳定不变量
- 把平台规则和物理学科规则分开

理由：

- 这是从“物理单仓”走向“平台仓”的第一步

### 目录 2

`prompts/shared/schemas/`

目的：

- 承接 manifest、rule、profile、override、snapshot 的协议校验

理由：

- 没有 schema，规则结构化会继续失控

### 目录 3

`tools/rule-compiler/`

目的：

- 承接规则加载、合并、profile 解析和 snapshot 编译

理由：

- 这是后续所有工具统一口径的中心

说明：

- `tools/answer-graphics/` 很重要，但不是首批 3 个目录之一
- 它应在规则快照体系稳定后再接入

## 6. 第一阶段实施包

### P0：资产与规则入口对齐

范围：

- `manifest`
- `config`
- `spec`
- `acceptance checklist`
- schema 骨架

验收：

- 资产层全部对齐 `v8.8`
- 没有 `v8.7` 漂移引用

### P1：快照编译器最小闭环

范围：

- 资产 schema 校验
- pack 加载
- profile 解析
- snapshot 输出

验收：

- 能生成第一版 `ResolvedSnapshot`

### P2：validator 接入 snapshot

范围：

- validator 改造
- evaluator 跟随 validator 走相同规则源

验收：

- validator 不再直接依赖散落 config

### P3：deliver 接入 snapshot

范围：

- deliver 编译并记录快照
- render 先保留兼容入口

验收：

- 一次交付能记录 `snapshotId`

## 10. 暂缓事项

当前不进入首轮优先级的内容：

- 第二学科包实际落地
- 高复杂度数学几何自动作图
- 完整桌面编辑器
- 全量 OCR provider 抽象重写

原因：

- 这些工作都依赖快照体系先稳定

## 11. 推荐执行周序

若按紧凑执行节奏，建议：

- 第 1 周：P0 + P1
- 第 2 周：P2 + P3
- 第 3 周：P4 + P5
- 第 4 周：P6 + P7
- 第 5 周：P8 + P9
- 第 6 周以后：P10 + P11 + P12

## 12. 一句话结论

最先要做的不是桌面壳，也不是作图题自动化，而是：

- 先对齐 `v8.8`
- 先建 `platform-core / shared schemas / rule-compiler`
- 先让 validator 和 deliver 跑在统一 snapshot 上

只有这一步稳住，后面的作图题答案图、PDF 排版、跨学科扩展才不会持续返工。
