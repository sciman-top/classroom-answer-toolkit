# 首版目录改造方案

## 1. 文档定位

本文档是 [工程终态与平台化路线图.md](D:/CODE/classroom-answer-toolkit/工程终态与平台化路线图.md) 的首版落地目录方案。

目标：

- 在不推倒现有仓库的前提下
- 通过最小目录重组
- 让平台核心、学科包、评测、规则快照、作图题子系统有明确归宿

原则：

- 只做必要目录升级
- 不大规模迁移现有稳定工具
- 不引入第二套平行资产根
- 先搭骨架，再逐步搬迁内容

## 2. 当前目录问题

当前主要目录：

- `prompts/physics-answer/`
- `tools/latex-renderer/`
- `tools/ocr/`
- `eval/physics-answer/`

主要问题：

- 只有学科包，没有平台核心资产层
- 没有共享 schema 层
- 没有规则编译器目录
- 作图题答案图子系统没有归宿
- 工具层目前仍偏单体脚本集合

## 3. 目录改造目标

首版目录改造后，仓库应形成四个稳定区：

1. 规则与资产区
2. 工具与编译区
3. 评测与基线区
4. 业务输入与样例区

## 5. 目录调整说明

### 5.1 prompts

#### `prompts/platform-core/`

新增。

职责：

- 平台稳定不变量
- 默认 profile 基线
- 通用交付规则
- 通用视觉审阅规则

首版先只放最关键规则，不追求丰富。

#### `prompts/physics-answer/`

保留，并从“单一资产目录”升级为“第一个正式 subject pack”。

新增子目录：

- `rules/`
- `profiles/`
- `overrides/`
- `migrations/`

首版改造原则：

- `manifest.json`、`config.json`、`spec.md`、`checklists/acceptance.md` 保留原位
- 新结构先增量引入，不先删旧文件

#### `prompts/shared/schemas/`

新增。

职责：

- 存放资产协议 schema
- 为后续 rule compiler 和 validator 提供统一校验依据

### 5.2 tools

#### `tools/rule-compiler/`

新增。

这是首版目录改造的关键。

职责：

- 校验资产
- 合并规则
- 解析 profile
- 编译 snapshot

要求：

- 只负责规则与快照
- 不负责具体 PDF 渲染

#### `tools/latex-renderer/`

保留。

职责不变，但输入源未来改为：

- 从 `runtime-config/config + profile` 逐步迁移到 `ResolvedSnapshot`

#### `tools/ocr/`

保留。

职责：

- OCR provider 实现
- 后续与平台级 `OcrProvider` contract 对齐

#### `tools/answer-graphics/`

新增。

职责：

- 原图裁剪
- 图形理解中间处理
- `AnswerGraphicSpec` 构建
- SVG 叠加层生成
- 合成预览图生成

首版不要求全部自动化，只要求形成独立归宿。

### 5.3 eval

`eval/physics-answer/` 保留。

新增建议：

- `baselines/graphics/`

用途：

- 保存作图题答案图专项视觉基线

### 5.4 artifacts

新增。

职责：

- 存放快照缓存
- 存放 review 临时产物
- 存放答案图中间产物

原因：

- 把中间态从业务输入目录中隔离出来
- 便于清理、复现和诊断

## 6. 首版不做的目录改造

首版明确不做：

- 不移动 `习题PDF/`
- 不将 `tools/latex-renderer/` 拆成多个 npm package
- 不引入 monorepo 工具链
- 不将 WPF 桌面壳目录提前塞入仓库

原因：

- 当前优先级是引擎平台化，而不是桌面壳搭建

## 7. 目录改造顺序

### 阶段 1

- 新增 `prompts/platform-core/`
- 新增 `prompts/shared/schemas/`
- 新增 `tools/rule-compiler/`
- 新增 `tools/answer-graphics/`
- 新增 `artifacts/`

### 阶段 2

- 在 `prompts/physics-answer/` 内补 `rules/`、`profiles/`、`overrides/`、`migrations/`
- 将现有 `tools/latex-renderer/profiles/*.json` 逐步转入学科包 profile 资产

### 阶段 3

- 将 validator / render / deliver 的规则入口逐步迁移到 `rule-compiler`
- 将作图题答案图专项评测接入 `eval/physics-answer/`

## 8. 与现有文件的映射关系

现有文件与新结构的映射建议：

- `prompts/physics-answer/config.json`
  - 保留，但逐步退化为汇总配置或兼容入口
- `prompts/physics-answer/manifest.json`
  - 升级为正式 `SubjectPackManifest`
- `prompts/physics-answer/spec.md`
  - 保留为人类可读镜像规范
- `tools/latex-renderer/profiles/classroom.json`
  - 未来迁到 `prompts/physics-answer/profiles/classroom.json`
- `tools/latex-renderer/profiles/compact.json`
  - 未来迁到 `prompts/physics-answer/profiles/compact.json`

## 9. 首版目录改造验收标准

目录改造完成后，至少满足：

- 平台核心资产层存在
- 学科包层结构完整
- 共享 schema 层存在
- rule compiler 有独立目录
- 作图题答案图子系统有独立目录
- 中间产物有统一暂存区
- 现有工具链仍能继续运行

## 10. 风险与回滚

风险：

- 目录先改了，但工具入口没改，导致双口径
- profile 迁移过早，影响现有渲染
- 中间产物目录过多，清理策略混乱

回滚：

- 首版只增目录，不删旧目录
- 所有旧脚本先保持可运行
- 等新入口稳定后，再逐步淘汰旧路径

## 11. 结论

首版目录改造的重点不是“好看”，而是尽快把：

- 平台层
- 学科层
- 快照层
- 作图题层

从当前较为平铺的仓库里显式分离出来，为后续模块化实现提供稳定落点。
